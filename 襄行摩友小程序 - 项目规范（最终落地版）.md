# 襄行摩友小程序 - 项目规范（最终落地版）

# 一、项目核心定位与合规红线（强制遵循）

## 1.1 核心定位

- **项目定位**：襄阳本地摩友公益服务工具，非盈利、纯工具属性，无交易、无社交、无导流。
- **运营模式**：小程序作为公益工具入口（展示活动记录、骑行路线、地图标注），活动组织通过微信群独立运营。
- **目标用户**：湖北省襄阳市本地摩托车骑行爱好者。
- **核心目标**：零成本开发、100%通过微信小程序审核，打造襄阳摩友日常骑行必备工具。

## 1.2 合规红线（绝对禁止）

- ❌ **禁止交易**：无支付、无电商、无二手买卖、无任何盈利性操作。
- ❌ **禁止社交**：无私聊、评论、点赞、关注、群聊、外部导流功能。
- ❌ **禁止展示联系方式**：不展示任何手机号、微信号、二维码、外链。
- ❌ **禁止收集敏感信息**：不收集手机号、身份证号、真实姓名等。
- ❌ **禁止AI深度合成**：不使用AI深度合成功能。
- ❌ **用户内容需审核**：用户上传内容（照片、互助信息等）必须经管理员人工审核后展示。
- ❌ **地图合规**：仅使用腾讯地图官方SDK，不私自标注未备案点位，不做商业广告。
- ✅ **合规声明**：小程序简介、首页、关于页必须明确标注"非盈利公益工具""摩友推荐非商业广告"等合规声明。

# 二、技术规范（匹配实际项目配置）

## 2.1 核心技术栈（严格匹配 package.json）

| 技术类别 | 具体选型 | 使用要求 |
|---------|---------|---------|
| 前端框架 | UniApp 3.0.0-4080520251106001 + Vue3.4.38 + TS5.5.4 + Vite5.2.8 | 统一使用 `<script setup lang="ts">`，禁止选项式API，TS类型全覆盖，禁止隐式any |
| UI组件库 | wot-design-uni 1.14.0 | 默认开启暗黑模式，统一使用原生组件，禁止自定义基础组件 |
| 开发辅助 | uni-helper 插件套件 | 启用组件自动引入、API自动类型提示 |
| 请求库 | alova 3.3.4 + @alova/adapter-uniapp | 统一放在 `src/api/` 目录，禁止裸写 `uni.request` |
| 状态管理 | Pinia 2.3.1 | 按模块划分状态，禁止全局变量滥用 |
| 地图服务 | 腾讯地图小程序SDK | 通过 `utils/tencentMap.ts` 封装调用，需用户授权 |
| 后端/存储 | Bmob后端云（免费版） | 通过 `utils/request.ts` 封装调用，预留密钥占位符 |
| 图表展示 | echarts 6.0.0 + uni-echarts 2.2.5 | 用于骑行数据可视化，配色贴合全局规范 |
| 包管理器 | pnpm 9.9.0 | 禁止使用npm、yarn，仅提交 pnpm-lock.yaml |

## 2.2 项目配置规范

- **package.json**：严格沿用现有配置，不随意升级、降级依赖。
- **vite.config.ts**：保留现有配置，优化构建速度。
- **pages.json**：配置所有页面路由、窗口样式（默认暗黑模式）、TabBar。
- **main.ts**：统一引入wot-design-uni、uni-helper、腾讯地图SDK，初始化Bmob。

## 2.3 目录规范（已落地，禁止修改结构）

```
src/
├── pages/           页面（index、map、data、activity、mine）
├── components/      业务组件（已按业务分离，自动引入）
├── composables/     自动导入组合式函数
├── hooks/           全局钩子
├── api/             alova 接口请求（统一封装）
├── utils/           工具函数（tencentMap、storage、request、route等）
├── store/           pinia 状态管理（按模块划分）
├── styles/          全局样式（含CSS变量，适配暗黑模式）
├── static/          静态资源（图标、图片、襄阳元素素材）
└── types/           全局TS类型定义
```

## 2.4 开发规范

- **组件规范**：组件自动引入已配置，无需手动import；命名用大驼峰。
- **样式规范**：100%使用UnoCSS原子类，禁止冗余style、全局样式污染，统一使用rpx单位。
- **语法规范**：异步逻辑统一使用async/await；小程序API统一使用`uni.xxx`。
- **条件编译**：使用 `#ifdef H5` / `#ifndef H5` 处理平台差异。

## 2.5 命令规范

```bash
pnpm dev:mp-weixin    # 开发微信小程序
pnpm build:mp-weixin  # 构建微信小程序
pnpm type-check       # 代码类型检查
pnpm lint:fix         # 代码格式化
```

# 三、功能规范

## 3.1 核心功能（5个Tab页）

### 首页（pages/index/index）
- 极简布局，顶部含襄阳定位+LOGO
- 快捷入口卡片（地图、数据、活动、工具）
- 公告栏滚动展示
- 骑行数据概览卡片
- 近期活动列表
- 底部合规声明

### 骑行地图（pages/map/map）
- 对接腾讯地图SDK
- 标注襄阳本地骑行相关点位（热门路线、打卡点、补给站）
- 实现定位、导航、筛选功能
- 热门路线推荐
- 路线详情展示

### 骑行数据（pages/data/data）
- 骑行数据可视化图表（月度里程、次数）
- 骑行统计卡片
- 骑行记录列表
- 海报生成功能（含襄阳元素）

### 活动工具（pages/activity/activity）
- **定位调整**：活动信息展示+回顾记录，非活动组织工具
- 活动列表展示（支持筛选：夜骑/晨骑/聚会/景区/绿道/免费）
- AA计算器（纯工具，无交易）
- 活动详情跳转地图路线
- 报名提示：引导用户关注群公告

### 我的页面（pages/mine/mine）
- 用户信息展示（预留Bmob用户系统）
- 骑行数据总览
- 功能入口（关于我们、帮助中心）
- 设置（暗黑模式切换、清除缓存）
- 襄阳特色展示
- **合规要求**：关于我们页面禁止展示任何联系方式

## 3.2 拓展功能（subPages分包）

| 功能 | 路径 | 说明 |
|-----|------|-----|
| 经典线路 | /subPages/route/route | 襄阳特色骑行线路，支持收藏、导航 |
| 骑行安全 | /subPages/safety/safety | 禁摩限行、应急知识、天气建议 |
| 工具合集 | /subPages/tools/tools | 油耗计算、保养提醒等高频工具 |
| 天气查询 | /subPages/weather/weather | 骑行天气参考 |
| 禁摩限行 | /subPages/traffic/traffic | 襄阳本地禁摩限行信息 |
| 物品互助 | /subPages/help/help | 物品赠送/互换，无价格、交易内容 |

## 3.3 活动更新规范

每次活动结束后，管理员需更新以下内容：

1. **活动记录**：在活动列表中更新活动状态为"已结束"
2. **地图标注**：如有新发现的优质路线/打卡点，添加到地图
3. **骑行数据**：活动参与者的骑行数据自动记录
4. **照片素材**：精选照片可用于首页轮播或路线详情

# 四、UI设计规范

## 4.1 设计原则

- **极简实用**：拒绝冗余设计，操作路径最短
- **沉浸式暗黑**：默认开启暗黑模式，所有页面适配
- **襄阳特色**：自然融入古城墙、汉江、古隆中等元素
- **一致性**：字体、颜色、组件样式统一

# 五、部署与审核规范

## 5.1 部署流程

1. 注册微信小程序（个人主体），完成实名认证，获取AppID
2. 在微信开发者工具中导入项目，配置AppID
3. 调试Bmob、腾讯地图接口，确保正常运行
4. 测试无误后，提交小程序审核（类目选择"工具-出行/旅游"）
5. 审核通过后发布上线

## 5.2 审核注意事项

- **合规声明**：小程序简介、首页、关于页必须包含"非盈利""非商业广告""无交易社交"声明
- **内容审核**：确认所有用户上传内容均有审核机制
- **功能排查**：确保无禁止功能，地图使用正规SDK
- **联系方式**：禁止在小程序任何位置展示微信号、手机号、二维码
